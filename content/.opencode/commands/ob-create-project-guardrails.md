Analyze `ARCHITECTURE.md` and other project files to generate a `project-guardrails` skill ŌĆö a set of rules and constraints extracted from the project's own documentation that agents must follow.

**Steps**

1. **Read source documents**

   Read ALL of the following that exist:
   - `ARCHITECTURE.md` (primary source)
   - `DESIGN.md` (design system, component conventions)
   - `AGENTS.md` (existing agent instructions, optimizations)
   - `README.md` (setup, conventions)
   - `CONTRIBUTING.md` (if present)
   - `.opencode/opencode-onboard.json` (platform, models, concurrency)
   - `openspec/config.yaml` (if present ŌĆö domain context and rules)
   - Root config files: `package.json`, `tsconfig.json`, `biome.json`, `.eslintrc*`, `Cargo.toml`, `go.mod`, `pyproject.toml`, `pom.xml` ŌĆö whatever exists
   - CI/CD workflows: `.github/workflows/*`, `azure-pipelines.yml` ŌĆö whatever exists

   Do not rely on prior knowledge ŌĆö read the actual files.

2. **Extract guardrails**

   From the documents above, extract concrete, actionable rules in these categories. Only include a category if you found real evidence for it:

   - **Architecture constraints** ŌĆö layer boundaries, module dependencies, forbidden imports, directory ownership rules (e.g. "src/api/ must not import from src/ui/")
   - **Naming conventions** ŌĆö file naming, component naming, API route conventions, branch naming
   - **Code style** ŌĆö formatter config, lint rules, import ordering, max line length ŌĆö derive from actual config files, not guesses
   - **Testing rules** ŌĆö test file locations, naming, coverage gates, what must be tested before merge
   - **Build & deployment** ŌĆö build commands, env requirements, deployment targets, CI gates
   - **Data & state** ŌĆö migration rules, schema change process, state management patterns
   - **Security** ŌĆö auth boundaries, input validation requirements, secrets handling
   - **Dependencies** ŌĆö package manager, lockfile rules, upgrade policy, forbidden packages
   - **Git workflow** ŌĆö branch naming, commit message format, PR process (platform-specific from `opencode-onboard.json`)
   - **Domain-specific rules** ŌĆö anything in `openspec/config.yaml` context or `ARCHITECTURE.md` constraints/risks sections

   Each rule must be:
   - **Concrete** ŌĆö "Use `pnpm` not `npm`" not "Use the right package manager"
   - **Evidence-based** ŌĆö derive from the files you read, do not invent rules
   - **Actionable** ŌĆö an agent can check it before acting

3. **Write the skill**

   Create `.agents/skills/project-guardrails/SKILL.md`:

   ```markdown
   ---
   name: project-guardrails
   description: Project-specific rules and constraints extracted from ARCHITECTURE.md. Load this skill before implementing any change to understand boundaries, conventions, and constraints for this codebase.
   license: MIT
   ---

   # Project Guardrails

   > Auto-generated by `/ob-create-project-guardrails`. Regenerate with the same command when architecture or conventions change.

   ## Architecture Constraints
   - <rule>
   - <rule>

   ## Naming Conventions
   - <rule>

   ## Code Style
   - <rule>

   ## Testing
   - <rule>

   ## Build & Deployment
   - <rule>

   ## Data & State
   - <rule>

   ## Security
   - <rule>

   ## Dependencies
   - <rule>

   ## Git Workflow
   - <rule>
   ```

   Only include sections that have real rules. Omit empty sections.

4. **Update agents**

   For every `*-engineer.md` in `.opencode/agents/`, add `@project-guardrails` to the Guardrails ability line:

   ```markdown
   ## Abilities
   - Guardrails: @ob-generic-guardrails, @project-guardrails, @ob-default
   ```

   If `@project-guardrails` is already present, skip that agent.

5. **Report**

   Tell the user:
   - Skill created at `.agents/skills/project-guardrails/SKILL.md`
   - Number of rules extracted per category
   - Number of agent files updated
   - Tip: "Rerun `/ob-create-project-guardrails` any time the architecture or conventions change significantly."
